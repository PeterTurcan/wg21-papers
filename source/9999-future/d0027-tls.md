## Abstract

This paper asks the committee to advance transport security wrappers - `tls_context` for portable certificate management and `tls_stream` for encrypted I/O - as standard library vocabulary.

Every other language has this. C++ does not. Go has `crypto/tls`. Rust has `rustls` and `native-tls`. Python has `ssl`. .NET has `SslStream`. Java has `SSLSocket`. Each wraps TLS the same way: a configuration object and an encrypted stream. The wrapper API has been stable since SSLv3 in 1996. The interface carries no risk. The cryptographic engine is implementation-defined - the same model as `std::filesystem`.

A `tls_stream` wraps any `Stream` and satisfies `Stream` itself. Certificate management is portable. The engine is not. A TLS vulnerability disclosed on Monday is patchable by Tuesday. The fix does not wait for a standard library release cycle.

The companion paper *TLS: Design Rationale*<sup>[9]</sup> provides the design rationale, the convergence record across seven ecosystems, anticipated objections, and the implementation inventory. Read this paper for the proposal; read the companion when you need the audit trail.

This paper is Paper 14 in the series defined by [P4100R1](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2026/p4100r1.pdf)<sup>[1]</sup>. It depends on Paper 1 (the IoAwaitable Protocol, [P4003R3](https://isocpp.org/files/papers/P4003R3.pdf)<sup>[2]</sup> and [P4172R0](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2026/p4172r0.pdf)<sup>[3]</sup>) and Paper 6 (Stream Concepts, D0011/D0012). Stage Two paper. Final paper in the series.

## 2. What This Paper Asks

The committee is asked to advance the following vocabulary for the standard library:

```cpp
namespace std::io {

  enum class tls_method { tlsv12, tlsv13 };
  enum class handshake_type { client, server };
  enum class verify_mode : unsigned {
    none           = 0,
    peer           = 1,
    fail_if_no_peer_cert = 2
  };
  enum class file_format { pem, asn1 };

  class tls_context;

  template<Stream NextLayer>
  class tls_stream;

}
```

Two enumerations for protocol version and handshake role. One enumeration for verification policy. One enumeration for certificate file format. One configuration class. One stream template. The same buffer sequences and error model as every other stream in the series.

## 3. `tls_context`

### 3.1 Purpose

`tls_context` is the configuration object. It holds certificates, private keys, trust anchors, and verification policy. It is constructed once and shared across connections. A server creates one `tls_context` with its certificate and key. A client creates one `tls_context` with the system trust store. Both pass the context by reference to every `tls_stream` they construct.

### 3.2 Interface

```cpp
class tls_context
{
public:
    explicit tls_context(tls_method method);

    tls_context(tls_context&&) noexcept;
    tls_context& operator=(tls_context&&) noexcept;
    ~tls_context();

    tls_context(tls_context const&) = delete;
    tls_context& operator=(tls_context const&) = delete;

    void use_certificate_file(
        std::filesystem::path const& p,
        file_format fmt);

    void use_private_key_file(
        std::filesystem::path const& p,
        file_format fmt);

    void use_certificate_chain_file(
        std::filesystem::path const& p);

    void set_default_verify_paths();

    void set_verify_mode(verify_mode mode);
};
```

### 3.3 Certificate Loading

Three functions load cryptographic material from files:

**`use_certificate_file`** loads a single certificate. The certificate identifies the local endpoint to the remote peer. Servers always need one. Clients need one only for mutual TLS.

**`use_private_key_file`** loads the private key corresponding to the certificate. The key proves ownership of the certificate.

**`use_certificate_chain_file`** loads a chain of intermediate certificates. The chain allows the remote peer to build a trust path from the endpoint certificate to a root trust anchor.

All three accept a `std::filesystem::path` and a `file_format`. PEM and ASN.1 (DER) are the two formats every TLS engine supports.

```cpp
std::io::tls_context ctx(tls_method::tlsv13);
ctx.use_certificate_chain_file(
    "/etc/ssl/server-chain.pem");
ctx.use_certificate_file(
    "/etc/ssl/server.pem", file_format::pem);
ctx.use_private_key_file(
    "/etc/ssl/server-key.pem", file_format::pem);
```

### 3.4 Trust Anchors

```cpp
void set_default_verify_paths();
```

Loads the operating system's default trust store. On Linux, this is typically `/etc/ssl/certs`. On macOS, the system Keychain. On Windows, the certificate store. The function delegates to the implementation-defined engine's platform integration.

A client that needs to verify server certificates calls `set_default_verify_paths()`. No paths to manage. No CA bundles to ship.

```cpp
std::io::tls_context ctx(tls_method::tlsv13);
ctx.set_default_verify_paths();
ctx.set_verify_mode(verify_mode::peer);
```

### 3.5 Verification Mode

```cpp
void set_verify_mode(verify_mode mode);
```

| Mode                   | Effect                                                     |
| ---------------------- | ---------------------------------------------------------- |
| `none`                 | Do not request or verify a peer certificate                |
| `peer`                 | Request and verify the peer certificate                    |
| `fail_if_no_peer_cert` | Fail the handshake if the peer does not present a certificate (server-side; combined with `peer`) |

The default is `none`. A TLS client that connects to the public internet should set `verify_mode::peer`. A TLS server that requires mutual authentication should set `verify_mode::peer | verify_mode::fail_if_no_peer_cert`.

## 4. `tls_stream`

### 4.1 Purpose

`tls_stream` wraps any type that satisfies `Stream` and provides encrypted I/O over it. The wrapped stream is the transport. The `tls_stream` is the encrypted channel. The most common transport is `tcp_socket`, but any `Stream` works - including `any_stream`, a UNIX domain socket, or another `tls_stream` (nested TLS).

### 4.2 Interface

```cpp
template<Stream NextLayer>
class tls_stream
{
public:
    tls_stream(NextLayer&& next, tls_context& ctx);

    tls_stream(tls_stream&&) noexcept;
    tls_stream& operator=(tls_stream&&) noexcept;
    ~tls_stream();

    tls_stream(tls_stream const&) = delete;
    tls_stream& operator=(tls_stream const&) = delete;

    IoAwaitable auto handshake(handshake_type type);

    IoAwaitable auto read_some(
        MutableBufferSequence auto const& buffers);

    IoAwaitable auto write_some(
        ConstBufferSequence auto const& buffers);

    IoAwaitable auto shutdown();

    NextLayer& next_layer() noexcept;
    NextLayer const& next_layer() const noexcept;
};
```

### 4.3 Satisfies `Stream`

`tls_stream` provides `read_some()` and `write_some()` with the same signatures as `tcp_socket`, `stream_file`, and every other `Stream` in the series. It satisfies `ReadStream`, `WriteStream`, and `Stream`. Any function that accepts `any_stream&` handles encrypted and unencrypted connections without change.

```cpp
std::io::task<> serve(std::io::any_stream& stream)
{
    // Does not know or care whether the stream
    // is encrypted. Same code. Same algorithms.
    auto [ec, n] = co_await stream.read_some(
        std::io::buffer(buf));
}
```

### 4.4 Layering

The `tls_stream` sits between the transport and the application. The layering is explicit in the type:

```cpp
std::io::tcp_socket sock(ctx, endpoint);
co_await sock.connect(endpoint);

std::io::tls_stream<std::io::tcp_socket> tls(
    std::move(sock), tls_ctx);
co_await tls.handshake(handshake_type::client);

// Application reads and writes through tls.
auto [ec, n] = co_await tls.read_some(
    std::io::buffer(buf));
```

After `handshake()` completes, `read_some()` and `write_some()` encrypt and decrypt transparently. The application never touches the raw transport again. Access to the underlying transport is available through `next_layer()` for operations that do not involve TLS - connection metadata, socket options, shutdown sequencing.

## 5. Handshake

### 5.1 Client vs Server

```cpp
IoAwaitable auto handshake(handshake_type type);
```

The `handshake_type` argument determines the role:

| Type       | Role                                                          |
| ---------- | ------------------------------------------------------------- |
| `client`   | Initiates the TLS handshake. Sends ClientHello                |
| `server`   | Waits for the client to initiate. Responds to ClientHello     |

The handshake is an asynchronous multi-round-trip protocol. The awaitable suspends the coroutine until the handshake completes or fails. On success, the stream is ready for encrypted I/O. On failure, the error code indicates the cause - certificate verification failure, protocol mismatch, or transport error.

```cpp
auto [ec] = co_await tls.handshake(
    handshake_type::client);
if (ec)
{
    // Handshake failed. ec describes why.
    co_return ec;
}
```

### 5.2 Server Name Indication

SNI (Server Name Indication) allows a client to indicate the hostname it is connecting to during the handshake. This is essential for virtual hosting - multiple domains served from a single IP address. The server uses the SNI hostname to select the correct certificate.

SNI configuration is part of the `tls_context` or the handshake initiation, depending on the implementation. The mechanism is implementation-defined. The requirement is not: a conforming implementation must support SNI.

### 5.3 ALPN Negotiation

Application-Layer Protocol Negotiation allows the client and server to agree on a protocol (HTTP/1.1, h2, etc.) during the handshake. ALPN avoids an extra round trip that would be required to negotiate the protocol after the handshake completes.

ALPN configuration is implementation-defined. The requirement is that a conforming implementation must support ALPN when the underlying engine provides it.

## 6. Certificate Verification

Certificate verification is the process by which the TLS implementation determines whether a peer's certificate is trustworthy. The process involves:

1. **Chain building.** Constructing a chain from the peer certificate through intermediate certificates to a root trust anchor.
2. **Signature validation.** Verifying each signature in the chain.
3. **Expiration checking.** Confirming no certificate in the chain has expired.
4. **Revocation checking.** Consulting CRLs or OCSP responders when available.
5. **Hostname matching.** Confirming the server certificate matches the requested hostname.

Steps 1-4 are performed by the implementation-defined engine. Step 5 is performed by the implementation when the application configures a hostname for verification. The application controls the policy through `set_verify_mode()`. The engine performs the cryptographic work.

The standard specifies that verification must occur when `verify_mode::peer` is set. The standard does not specify the cryptographic algorithms used to verify signatures. Those are the engine's responsibility.

---

## 7. The Cryptographic Engine Is Implementation-Defined

The standard does not mandate OpenSSL, WolfSSL, BoringSSL, SChannel, Secure Transport, or any other specific TLS engine. The cryptographic engine is implementation-defined.

This is the `std::filesystem` model. `std::filesystem::remove()` does not specify whether it calls `unlink()` or `DeleteFileW()`. It specifies the effect. The implementation provides the mechanism. A filesystem vulnerability in the kernel is patchable without recompiling user code. A TLS vulnerability in the engine is patchable without recompiling user code.

The alternative - mandating a specific engine - creates three problems:

1. **Vendor lock-in.** Implementations on platforms with native TLS stacks (Windows SChannel, macOS Secure Transport) would be forced to ship and maintain a second engine.
2. **Update lag.** A mandated engine becomes the committee's maintenance burden. A CVE in the mandated engine becomes a standard library defect.
3. **Regulatory conflict.** Some jurisdictions require the use of specific, certified cryptographic implementations. Mandating an engine would make compliance impossible.

The implementation-defined model avoids all three.

**The interface is portable. The engine is not. That is the point.**
