## Abstract

This paper asks the committee to advance a datagram socket type for coroutine-native I/O.

UDP is the transport beneath DNS, QUIC/HTTP/3, game networking, media streaming, and IoT protocols. The type proposed here - `std::io::udp_socket` - provides `send_to`, `receive_from`, `bind`, multicast group management, and an optional connected mode. It ships today in [Corosio](https://github.com/cppalliance/corosio)<sup>[1]</sup> on three platforms. The proposal preserves datagram message boundaries, does not impose a stream abstraction, and integrates with the IoAwaitable protocol ([P4003R3](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2026/p4003r3.pdf)<sup>[2]</sup>).

The companion paper *UDP: Design Rationale*<sup>[3]</sup> provides the platform mapping, the convergence record across six ecosystems, anticipated objections, and the implementation inventory. Read this paper for the proposal; read the companion when you need the audit trail.

This paper is Paper 13 in the series defined by [P4100R1](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2026/p4100r1.pdf)<sup>[4]</sup>. It depends on Paper 1 (IoAwaitable Protocol, [P4003R3](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2026/p4003r3.pdf)<sup>[2]</sup>) and Paper 11 (TCP, for `endpoint` and `ip_address` types). It is a Stage Two paper.

## 2. What This Paper Asks

The committee is asked to advance the following type for the standard library:

```cpp
namespace std::io {

  class udp_socket {
  public:
    explicit udp_socket(execution_context& ctx);

    udp_socket(udp_socket const&) = delete;
    udp_socket& operator=(udp_socket const&) = delete;
    udp_socket(udp_socket&&) noexcept;
    udp_socket& operator=(udp_socket&&) noexcept;

    ~udp_socket();

    // Binding
    void bind(endpoint const& ep);
    void close();

    // Unconnected datagram I/O
    IoAwaitable auto send_to(
        ConstBufferSequence auto const& buffers,
        endpoint const& dest);
    IoAwaitable auto receive_from(
        MutableBufferSequence auto const& buffers,
        endpoint& sender);

    // Connected UDP
    IoAwaitable auto connect(endpoint const& ep);
    IoAwaitable auto send(
        ConstBufferSequence auto const& buffers);
    IoAwaitable auto receive(
        MutableBufferSequence auto const& buffers);

    // Multicast
    void join_group(ip_address const& multicast_addr);
    void leave_group(ip_address const& multicast_addr);
  };

}
```

One type. Two I/O modes. Two multicast operations. One bind. One close.

## 3. The `udp_socket` Type

### 3.1 Construction

```cpp
explicit udp_socket(execution_context& ctx);
```

A `udp_socket` is constructed with a reference to an `execution_context` - the event loop that will deliver completions. The socket starts unbound and unconnected. The execution context must outlive the socket.

```cpp
std::io::execution_context ctx;
std::io::udp_socket sock(ctx);
```

### 3.2 `bind()`

```cpp
void bind(endpoint const& ep);
```

Binds the socket to a local address and port. The socket must be bound before calling `receive_from` or joining a multicast group. The endpoint specifies the local interface and port; a wildcard address (`0.0.0.0` or `::`) binds to all interfaces.

```cpp
std::io::endpoint local(
    std::io::ipv4_address::any(), 9000);
sock.bind(local);
```

Binding is a synchronous operation. It configures the kernel association between the socket and the local address. No I/O is initiated.

### 3.3 `send_to()` and `receive_from()`

```cpp
IoAwaitable auto send_to(
    ConstBufferSequence auto const& buffers,
    endpoint const& dest);

IoAwaitable auto receive_from(
    MutableBufferSequence auto const& buffers,
    endpoint& sender);
```

The primary datagram operations. `send_to` transmits the contents of `buffers` as a single datagram to the destination endpoint. `receive_from` receives a single datagram into `buffers` and writes the sender's address into `sender`.

Both return awaitables satisfying the IoAwaitable protocol. The result type for both is `io_result<std::size_t>` - an error code and the number of bytes transferred.

```cpp
std::io::endpoint sender;
char buf[65536];

auto [ec, n] = co_await sock.receive_from(
    std::io::buffer(buf), sender);
if (!ec)
    std::println("received {} bytes from {}",
        n, sender);
```

**Message boundaries are preserved.** Each `send_to` produces exactly one datagram. Each `receive_from` delivers exactly one datagram. If the receive buffer is smaller than the datagram, the excess bytes are discarded and the operation succeeds with the truncated size. The companion design paper<sup>[3]</sup> Section 3 documents why this differs from TCP stream semantics and why no stream abstraction is imposed.

### 3.4 `close()`

```cpp
void close();
```

Closes the socket and releases the underlying file descriptor. Any outstanding asynchronous operations complete with `std::errc::operation_canceled`. After `close()`, the socket is in a moved-from state.

## 4. Datagram Semantics

A UDP socket is not a stream. TCP delivers a byte stream with no message boundaries. UDP delivers discrete datagrams. Each `send_to` produces one datagram. Each `receive_from` delivers one datagram. The kernel does not coalesce, split, or reorder within a single operation.

`udp_socket` does not satisfy `Stream` (Paper 6). Stream concepts assume a byte-oriented, potentially infinite sequence - `read_some` may return a partial result that the caller must reassemble. Datagrams have no partial result. The entire message arrives in one operation or it does not arrive at all.

The consequence for the API is that `udp_socket` has no `read_some` or `write_some` member functions. The operations are `send_to`/`receive_from` (unconnected) and `send`/`receive` (connected). The naming reflects the semantic difference: the destination is part of each send operation, the source is part of each receive operation.

## 5. Multicast

### 5.1 `join_group()`

```cpp
void join_group(ip_address const& multicast_addr);
```

Joins a multicast group. The socket must be bound before joining. After joining, datagrams sent to the multicast address are delivered to this socket. The implementation issues `IP_ADD_MEMBERSHIP` (IPv4) or `IPV6_JOIN_GROUP` (IPv6) on the underlying file descriptor.

### 5.2 `leave_group()`

```cpp
void leave_group(ip_address const& multicast_addr);
```

Leaves a previously joined multicast group. After leaving, datagrams sent to the multicast address are no longer delivered. If the socket is closed or destroyed, all group memberships are implicitly dropped.

### 5.3 Usage

```cpp
std::io::endpoint local(
    std::io::ipv4_address::any(), 5353);
sock.bind(local);

std::io::ipv4_address mcast_addr("239.255.255.250");
sock.join_group(mcast_addr);

std::io::endpoint sender;
char buf[4096];
auto [ec, n] = co_await sock.receive_from(
    std::io::buffer(buf), sender);
```

Multicast is the mechanism behind mDNS, SSDP/UPnP, PTP time synchronisation, and real-time media distribution. The two-function interface - `join_group` and `leave_group` - is the minimal surface that covers these use cases. Advanced multicast options (source-specific multicast, TTL control, loopback control) are documented as deferrals in the companion design paper<sup>[3]</sup> Section 9.

## 6. Connected UDP

UDP sockets may optionally be connected to a remote endpoint. After connection, `send` and `receive` operate without specifying a destination or receiving a source address. The kernel filters incoming datagrams to only those from the connected peer.

### 6.1 `connect()`

```cpp
IoAwaitable auto connect(endpoint const& ep);
```

Associates the socket with a remote endpoint. After a successful `connect`, the socket accepts `send`/`receive` without per-call endpoint arguments. On most platforms, `connect` on a UDP socket is a local kernel operation that does not send any network traffic.

### 6.2 `send()` and `receive()`

```cpp
IoAwaitable auto send(
    ConstBufferSequence auto const& buffers);
IoAwaitable auto receive(
    MutableBufferSequence auto const& buffers);
```

Connected-mode I/O. `send` transmits to the connected endpoint. `receive` accepts only datagrams from the connected endpoint - the kernel discards datagrams from other sources.

The result type for both is `io_result<std::size_t>`.

```cpp
std::io::endpoint server(
    std::io::ipv4_address("192.168.1.1"), 53);
co_await sock.connect(server);

co_await sock.send(std::io::buffer(query));
auto [ec, n] = co_await sock.receive(
    std::io::buffer(response));
```

### 6.3 When to Use Connected UDP

Connected UDP is appropriate when all traffic flows to and from a single peer - DNS stub resolvers, QUIC connections, game client sessions. The kernel-level filtering reduces per-packet overhead and prevents spoofed-source datagrams from reaching the application. The companion design paper<sup>[3]</sup> Section 4 provides the design rationale for supporting both modes.

## 7. Example Usage

### 7.1 Echo Server

```cpp
task<> udp_echo(std::io::execution_context& ctx)
{
    std::io::udp_socket sock(ctx);
    std::io::endpoint local(
        std::io::ipv4_address::any(), 7);
    sock.bind(local);

    char buf[65536];
    for (;;)
    {
        std::io::endpoint sender;
        auto [ec, n] = co_await sock.receive_from(
            std::io::buffer(buf), sender);
        if (ec)
            co_return;

        co_await sock.send_to(
            std::io::buffer(buf, n), sender);
    }
}
```

### 7.2 DNS Query (Connected Mode)

```cpp
task<std::vector<std::byte>>
dns_query(std::io::execution_context& ctx,
          std::span<std::byte const> query)
{
    std::io::udp_socket sock(ctx);
    std::io::endpoint dns_server(
        std::io::ipv4_address("8.8.8.8"), 53);
    co_await sock.connect(dns_server);

    co_await sock.send(std::io::buffer(query));

    std::vector<std::byte> response(512);
    auto [ec, n] = co_await sock.receive(
        std::io::buffer(response));
    if (ec)
        throw std::system_error(ec);
    response.resize(n);
    co_return response;
}
```

### 7.3 Multicast Listener

```cpp
task<> ssdp_listener(std::io::execution_context& ctx)
{
    std::io::udp_socket sock(ctx);
    std::io::endpoint local(
        std::io::ipv4_address::any(), 1900);
    sock.bind(local);

    sock.join_group(
        std::io::ipv4_address("239.255.255.250"));

    char buf[4096];
    for (;;)
    {
        std::io::endpoint sender;
        auto [ec, n] = co_await sock.receive_from(
            std::io::buffer(buf), sender);
        if (ec)
            break;
        process_ssdp(
            std::string_view(buf, n), sender);
    }
}
```
