## Abstract

This paper asks EWG to add `int_least128_t`, `uint_least128_t`, `int_fast128_t`, and `uint_fast128_t` to `<cstdint>` as required extended integer type aliases, with `int128_t` and `uint128_t` as optional exact-width aliases. These are extended integer types, not standard integer types. No core language changes are required.

---

This paper does not compete with P3666. It addresses the narrower problem of portable 128-bit integers on every major compiler.

## 3. Design

### 3.1 Types Added

| Type | Width | Required |
|---|---|---|
| `int_least128_t` / `uint_least128_t` | >= 128 bits | mandatory |
| `int_fast128_t` / `uint_fast128_t` | >= 128 bits | mandatory |
| `int128_t` / `uint128_t` | exactly 128 bits | optional |

The exact-width types `int128_t` / `uint128_t` are provided if and only if the implementation supports a signed integer type with exactly 128 value bits and no padding. This follows the existing pattern for `int8_t` through `int64_t`.

### 3.2 Extended Integer Types

The 128-bit types are extended integer types, not standard integer types. This is consistent with [basic.fundamental] which permits implementation-defined extended signed and unsigned integer types.<sup>[9]</sup>

Extended integer types already participate in:

- Integer conversion rank ([conv.rank]) - wider types rank higher<sup>[9]</sup>
- `numeric_limits` - mandatory for all arithmetic types
- `make_signed` / `make_unsigned` - fallback rule selects matching type by size
- `to_chars` / `from_chars` - accept all integer types
- `format` - accept all integer types

No wording changes are needed for these facilities.

### 3.3 `intmax_t` Is Not Affected

[LWG 3828](https://timsong-cpp.github.io/lwg-issues/3828)<sup>[2]</sup>, adopted in C++23, decoupled `intmax_t` from being the widest integer type. Extended integer types wider than `long long` are excluded from `intmax_t`'s coverage. This was the historical blocker that prevented implementations from exposing `__int128` as a conforming extended integer type.<sup>[3]</sup> The blocker no longer exists.

### 3.4 No Core Language Changes

The 128-bit types are extended integer types. [basic.fundamental] already permits them. [conv.rank] already ranks them. No grammar productions, no new keywords, no changes to the type system. The work is entirely in the library.

### 3.5 C Compatibility

C23 adopted `%w128d` and `%wf128d` format specifiers via [N2680](https://www.open-std.org/JTC1/SC22/WG14/www/docs/n2680.pdf)<sup>[4]</sup>, providing `printf` support for 128-bit types. SG22 reviewed [P3140R1](https://eisenwave.github.io/cpp-proposals/int-least128.html)<sup>[5]</sup> (the prior proposal for `int_least128_t`) and found no C/C++ compatibility concerns. WG14 expressed interest in a corresponding C paper.<sup>[6]</sup>

### 3.6 Relationship to P3666

[P3666R3](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2026/p3666r3.html)<sup>[1]</sup> proposes adopting C23's `_BitInt(N)` into C++. The two proposals are not mutually exclusive. `_BitInt(128)` and `__int128_t` are distinct types with distinct mangling (`DB128_` vs `n`/`o` in the Itanium ABI<sup>[7]</sup>) and distinct alignment (8 bytes vs 16 bytes on x86-64<sup>[8]</sup>). Standardizing `_BitInt` does not retire `__int128_t`. This paper standardizes the type that exists.

## 4. Proposed Wording

Relative to [N5032](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2025/n5032.pdf)<sup>[9]</sup>.

The wording follows the structure of [P3140R1](https://eisenwave.github.io/cpp-proposals/int-least128.html)<sup>[5]</sup>.

### 4.1 Sections Requiring No Wording Changes

The following sections already handle extended integer types and require no modification:

- [basic.fundamental] - extended integer types are already permitted
- [conv.rank] - ranking rules already cover extended types by width
- [meta.trans.sign] - `make_signed` / `make_unsigned` already select by size
- [charconv] - `to_chars` / `from_chars` already accept integer types
- [format.arg] - formatting already accepts integer types
- [numeric.limits] - specialization is mandatory for all arithmetic types

### 4.2 [cstdint.syn]

:::wording

Add to the synopsis of header `<cstdint>`:

<ins>`int128_t`</ins>
<ins>`int_least128_t`</ins>
<ins>`int_fast128_t`</ins>
<ins>`uint128_t`</ins>
<ins>`uint_least128_t`</ins>
<ins>`uint_fast128_t`</ins>
<ins>`INT128_MIN`</ins>
<ins>`INT128_MAX`</ins>
<ins>`UINT128_MAX`</ins>
<ins>`INT_LEAST128_MIN`</ins>
<ins>`INT_LEAST128_MAX`</ins>
<ins>`UINT_LEAST128_MAX`</ins>
<ins>`INT_FAST128_MIN`</ins>
<ins>`INT_FAST128_MAX`</ins>
<ins>`UINT_FAST128_MAX`</ins>
<ins>`INT128_C`</ins>
<ins>`UINT128_C`</ins>

:::

### 4.3 [cstdint.general]

:::wording

Add after the existing description of exact-width, minimum-width, and fastest minimum-width types:

<ins>The typedef name `int_least128_t` designates a signed integer type with a width of at least 128 bits. The typedef name `uint_least128_t` designates the corresponding unsigned integer type. The typedef name `int_fast128_t` designates the fastest signed integer type with a width of at least 128 bits. The typedef name `uint_fast128_t` designates the corresponding unsigned integer type.</ins>

<ins>These types are extended integer types ([basic.fundamental]).</ins>

<ins>The typedef name `int128_t` designates a signed integer type with width exactly 128, no padding bits, and two's complement representation. The typedef name `uint128_t` designates the corresponding unsigned integer type. These exact-width types are optional. If an implementation provides a signed extended integer type with width exactly 128 and no padding bits, it provides `int128_t` and `uint128_t`.</ins>

<ins>The macros `INT128_MIN`, `INT128_MAX`, and `UINT128_MAX` are defined if and only if `int128_t` and `uint128_t` are defined. The macros `INT128_C` and `UINT128_C` are defined if and only if `int128_t` and `uint128_t` are defined.</ins>

<ins>The macros `INT_LEAST128_MIN`, `INT_LEAST128_MAX`, `UINT_LEAST128_MAX`, `INT_FAST128_MIN`, `INT_FAST128_MAX`, and `UINT_FAST128_MAX` have values corresponding to the properties of the types `int_least128_t`, `uint_least128_t`, `int_fast128_t`, and `uint_fast128_t`.</ins>

:::

### 4.4 [cinttypes.syn]

:::wording

Add to the synopsis of header `<cinttypes>`:

<ins>If `int128_t` is defined, the macros `PRId128`, `PRIi128`, `PRIo128`, `PRIu128`, `PRIx128`, `PRIX128`, `SCNd128`, `SCNi128`, `SCNo128`, `SCNu128`, `SCNx128` are defined with appropriate format specifier strings.</ins>

<ins>The macros `PRIdLEAST128`, `PRIiLEAST128`, `PRIoLEAST128`, `PRIuLEAST128`, `PRIxLEAST128`, `PRIXLEAST128`, `SCNdLEAST128`, `SCNiLEAST128`, `SCNoLEAST128`, `SCNuLEAST128`, `SCNxLEAST128` are defined with appropriate format specifier strings.</ins>

<ins>The macros `PRIdFAST128`, `PRIiFAST128`, `PRIoFAST128`, `PRIuFAST128`, `PRIxFAST128`, `PRIXFAST128`, `SCNdFAST128`, `SCNiFAST128`, `SCNoFAST128`, `SCNuFAST128`, `SCNxFAST128` are defined with appropriate format specifier strings.</ins>

:::

### 4.5 [version.syn]

:::wording

Add to Table 35 (Standard library feature-test macros):

<ins>`__cpp_lib_int128` with value `202XXXL`, header `<cstdint>`</ins>

:::

### 4.6 [template.bitset]

:::wording

Add to [bitset.cons]:

<ins>`constexpr bitset(int_least128_t val) noexcept;`</ins>
<ins>`constexpr bitset(uint_least128_t val) noexcept;`</ins>

<ins>*Effects*: Initializes the first M bit positions to the corresponding bit values in `val`. M is the smaller of N and 128. If M < N, the remaining bit positions are initialized to zero.</ins>

Add to [bitset.members]:

<ins>`template<class T> constexpr T to() const;`</ins>

<ins>*Constraints*: `T` is an integer type.</ins>

<ins>*Throws*: `overflow_error` if the integral value cannot be represented as a value of type `T`.</ins>

<ins>*Returns*: The integral value corresponding to the bits in `*this`.</ins>

:::

### 4.7 [string.conversions]

:::wording

Add overloads:

<ins>`string to_string(int_least128_t val);`</ins>
<ins>`string to_string(uint_least128_t val);`</ins>
<ins>`wstring to_wstring(int_least128_t val);`</ins>
<ins>`wstring to_wstring(uint_least128_t val);`</ins>

:::

### 4.8 [cmath.syn]

:::wording

Add overloads:

<ins>`int_least128_t abs(int_least128_t x);`</ins>

:::

### 4.9 [atomics.syn]

:::wording

Add type aliases:

<ins>`using atomic_int_least128_t  = atomic<int_least128_t>;`</ins>
<ins>`using atomic_uint_least128_t = atomic<uint_least128_t>;`</ins>
<ins>`using atomic_int_fast128_t   = atomic<int_fast128_t>;`</ins>
<ins>`using atomic_uint_fast128_t  = atomic<uint_fast128_t>;`</ins>

<ins>If `int128_t` is defined:</ins>

<ins>`using atomic_int128_t  = atomic<int128_t>;`</ins>
<ins>`using atomic_uint128_t = atomic<uint128_t>;`</ins>

:::

### 4.10 [ranges.iota.view]

:::wording

Modify the definition of *IOTA-DIFF-T(W)*:

<ins>If the width of `W` is not greater than 128, the type denoted by *IOTA-DIFF-T(W)* is a signed integer type of width greater than the width of `W` if such a type exists, otherwise `int_least128_t`.</ins>

:::

## 5. Suggested Straw Poll

**Poll.** EWG encourages a proposal to add mandatory `int_least128_t` / `uint_least128_t` and optional `int128_t` / `uint128_t` as extended integer types in `<cstdint>`.
