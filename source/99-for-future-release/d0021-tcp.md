## Abstract

This paper asks the committee to advance TCP sockets, acceptors, IP addresses, and endpoints as standard library vocabulary for coroutine-native networking.

The vocabulary is `tcp_socket`, `tcp_acceptor`, `endpoint`, `ipv4_address`, `ipv6_address`, and `ip_address`. A `tcp_socket` satisfies `Stream`. A `tcp_acceptor` binds, listens, and accepts connections that produce `tcp_socket` objects. The shapes are the [Networking TS](http://www.open-std.org/jtc1/sc22/wg21/docs/papers/2018/n4771.pdf)<sup>[1]</sup> shapes, deployed for over twenty years in [Boost.Asio](https://www.boost.org/doc/libs/release/doc/html/boost_asio.html)<sup>[2]</sup>, shipping today in [Corosio](https://github.com/cppalliance/corosio)<sup>[3]</sup>. This delivers what the committee has been trying to standardize since [N1925](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2005/n1925.pdf)<sup>[4]</sup> (2005).

The companion paper *TCP: Design Rationale*<sup>[10]</sup> provides the design rationale, the convergence record across six ecosystems, anticipated objections, and the implementation inventory. Read this paper for the proposal; read the companion when you need the audit trail.

This paper is Paper 11 in the series defined by [P4100R1](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2026/p4100r1.pdf)<sup>[5]</sup>. It depends on Paper 1 (IoAwaitable Protocol, [P4003R3](https://isocpp.org/files/papers/P4003R3.pdf)<sup>[6]</sup>) and Paper 6 (Stream Concepts). It is a Stage Two paper.

## 2. What This Paper Asks

The committee is asked to advance the following vocabulary for the standard library:

```cpp
namespace std::io {

  class ipv4_address;
  class ipv6_address;
  class ip_address;         // variant of ipv4/ipv6

  class endpoint;

  class tcp_socket;         // satisfies Stream
  class tcp_acceptor;

}
```

Three address types. One endpoint. One socket. One acceptor.

## 3. IP Addresses

### 3.1 `ipv4_address`

A 32-bit IPv4 address in network byte order.

```cpp
class ipv4_address
{
public:
    constexpr ipv4_address() noexcept;
    constexpr explicit ipv4_address(
        std::uint32_t addr) noexcept;
    constexpr ipv4_address(
        std::uint8_t a, std::uint8_t b,
        std::uint8_t c, std::uint8_t d) noexcept;

    constexpr std::uint32_t
        to_uint() const noexcept;

    std::string to_string() const;

    static result<ipv4_address>
        from_string(std::string_view s) noexcept;

    constexpr bool is_loopback() const noexcept;
    constexpr bool is_multicast() const noexcept;
    constexpr bool is_unspecified() const noexcept;

    friend constexpr bool operator==(
        ipv4_address const&,
        ipv4_address const&) noexcept = default;
    friend constexpr auto operator<=>(
        ipv4_address const&,
        ipv4_address const&) noexcept = default;

    static constexpr ipv4_address any() noexcept;
    static constexpr ipv4_address
        loopback() noexcept;
};
```

`from_string` returns a result type. Malformed input produces an error, not undefined behaviour. `to_string` renders the dotted-decimal form (`"192.168.1.1"`).

### 3.3 `ip_address`

A variant of `ipv4_address` and `ipv6_address`. Dual-stack servers accept connections from either protocol family and need a single address type that holds both.

```cpp
class ip_address
{
public:
    constexpr ip_address() noexcept;
    constexpr ip_address(
        ipv4_address const& addr) noexcept;
    constexpr ip_address(
        ipv6_address const& addr) noexcept;

    constexpr bool is_v4() const noexcept;
    constexpr bool is_v6() const noexcept;

    constexpr ipv4_address to_v4() const;
    constexpr ipv6_address to_v6() const;

    std::string to_string() const;

    static result<ip_address>
        from_string(std::string_view s) noexcept;

    constexpr bool is_loopback() const noexcept;
    constexpr bool is_multicast() const noexcept;
    constexpr bool is_unspecified() const noexcept;

    friend constexpr bool operator==(
        ip_address const&,
        ip_address const&) noexcept = default;
    friend constexpr auto operator<=>(
        ip_address const&,
        ip_address const&) noexcept = default;
};
```

## 4. Endpoints

An endpoint pairs an IP address with a port number. It is the addressing vocabulary for both TCP and UDP.

```cpp
class endpoint
{
public:
    constexpr endpoint() noexcept;
    constexpr endpoint(
        ip_address addr,
        std::uint16_t port) noexcept;

    constexpr ip_address
        address() const noexcept;
    constexpr void set_address(
        ip_address addr) noexcept;

    constexpr std::uint16_t
        port() const noexcept;
    constexpr void set_port(
        std::uint16_t port) noexcept;

    std::string to_string() const;

    friend constexpr bool operator==(
        endpoint const&,
        endpoint const&) noexcept = default;
    friend constexpr auto operator<=>(
        endpoint const&,
        endpoint const&) noexcept = default;
};
```

## 5. `tcp_socket`

A connected TCP socket. Satisfies `Stream` (Paper 6): provides `read_some` and `write_some` returning `IoAwaitable`. The central I/O type in the vocabulary.

```cpp
class tcp_socket
{
public:
    explicit tcp_socket(
        execution_context& ctx);

    tcp_socket(tcp_socket&& other) noexcept;
    tcp_socket& operator=(
        tcp_socket&& other) noexcept;

    tcp_socket(tcp_socket const&) = delete;
    tcp_socket& operator=(
        tcp_socket const&) = delete;

    ~tcp_socket();

    IoAwaitable auto connect(
        endpoint const& ep);

    IoAwaitable auto read_some(
        MutableBufferSequence auto buffers);

    IoAwaitable auto write_some(
        ConstBufferSequence auto buffers);

    void shutdown(shutdown_type what);
    void close();

    bool is_open() const noexcept;
    endpoint local_endpoint() const;
    endpoint remote_endpoint() const;

    void set_option(/* socket option */);
    void get_option(/* socket option */) const;

    native_handle_type native_handle();
};
```

### 5.1 Operations

**`connect(endpoint)`** initiates an asynchronous TCP connection to the given endpoint. Returns `IoAwaitable` yielding an `error_code`. On success the socket is connected and ready for I/O.

**`read_some(buffers)`** reads available data into the buffer sequence. Returns `IoAwaitable` yielding `io_result<std::size_t>` - an error code and a byte count. Zero bytes with no error indicates the peer closed the connection.

**`write_some(buffers)`** writes data from the buffer sequence. Returns `IoAwaitable` yielding `io_result<std::size_t>`. The operation may write fewer bytes than requested; the caller is responsible for retrying or using a composed `write` algorithm.

**`shutdown(what)`** shuts down one or both directions of the connection. `shutdown_type::send` sends a FIN; `shutdown_type::receive` closes the read half; `shutdown_type::both` closes both.

**`close()`** closes the underlying socket handle and cancels any pending asynchronous operations.

### 5.2 Socket Options

Socket options are set through a type-safe interface:

```cpp
tcp_socket sock(ctx);
co_await sock.connect(ep);

sock.set_option(tcp_nodelay(true));
sock.set_option(reuse_address(true));
sock.set_option(
    receive_buffer_size(65536));
sock.set_option(
    send_buffer_size(65536));
sock.set_option(keep_alive(true));
```

Each option type encapsulates the level and name constants. The set of standard option types includes at minimum `tcp_nodelay`, `reuse_address`, `keep_alive`, `receive_buffer_size`, `send_buffer_size`, and `linger`. The companion design paper<sup>[10]</sup> Section 6 discusses the option model.

## 6. `tcp_acceptor`

A TCP acceptor binds to a local endpoint, listens for incoming connections, and accepts them as connected `tcp_socket` objects.

```cpp
class tcp_acceptor
{
public:
    explicit tcp_acceptor(
        execution_context& ctx);

    tcp_acceptor(tcp_acceptor&& other) noexcept;
    tcp_acceptor& operator=(
        tcp_acceptor&& other) noexcept;

    tcp_acceptor(tcp_acceptor const&) = delete;
    tcp_acceptor& operator=(
        tcp_acceptor const&) = delete;

    ~tcp_acceptor();

    void bind(endpoint const& ep);
    void listen(
        int backlog = /* implementation-defined */);

    IoAwaitable auto accept();

    void close();
    bool is_open() const noexcept;

    endpoint local_endpoint() const;

    void set_option(/* socket option */);
    native_handle_type native_handle();
};
```

**`bind(endpoint)`** associates the acceptor with a local address and port. **`listen(backlog)`** marks the socket as passive and sets the connection backlog. **`accept()`** returns `IoAwaitable` yielding a connected `tcp_socket`. The canonical accept loop:

```cpp
std::io::task<> accept_loop(
    tcp_acceptor& acceptor)
{
    for (;;)
    {
        auto [ec, sock] =
            co_await acceptor.accept();
        if (ec)
            break;
        run_async(
            sock.get_executor()
        )(handle_connection(std::move(sock)));
    }
}
```

## 7. The Three-Layer Architecture

Every I/O type in the series exposes three API layers ([P4100R1](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2026/p4100r1.pdf)<sup>[5]</sup> Section 6). TCP sockets follow the same pattern:

```
      io_stream               abstract (Layer 3)
          |
     tcp_socket               concrete (Layer 2)
          |
native_tcp_socket<Backend>    native   (Layer 1)
```

**Abstract.** `io_stream` provides protocol-agnostic `read_some` and `write_some` with virtual dispatch. A library that accepts `any_stream&` works with TCP, TLS, or any future transport without recompilation.

**Concrete.** `tcp_socket` adds `connect`, `shutdown`, endpoint queries, and socket options. Still virtual dispatch. Separately compilable. This is the layer most application code uses.

**Native.** `native_tcp_socket<Backend>` is templated on the platform backend. Member function shadowing eliminates the vtable. Full inlining. For hot paths and benchmarks.

The user chooses the layer. Neither forces a choice on the other.
